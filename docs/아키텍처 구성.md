# 아키텍처 구성

이 프로젝트를 AWS에서 실제로 운영한다고 가정했을 때의 구성을 설명합니다.

## 구성도 (Mermaid)

```mermaid
graph TD
   UserAccount([사용자 계정])

   subgraph VPC[AWS VPC]
      subgraph PublicSubnet[Public Subnet]
         ALB[Application Load Balancer]
      end

      subgraph PrivateSubnetApp[Private Subnet - Application]
         subgraph EKS[Amazon EKS Cluster]
            subgraph ControlPlane[Control Plane]
               APIServer[K8s API Server]
            end

            subgraph WorkerNodes[Worker Nodes]
               subgraph NodeGroup[Managed Node Group]
                  subgraph Namespace[Namespace: default]
                     Service[Kubernetes Service]
                     Pod1[Spring Boot Pod #1]
                     Pod2[Spring Boot Pod #2]
                     Pod3[Spring Boot Pod #3]
                  end
               end
            end
         end
      end

      subgraph PrivateSubnetData[Private Subnet - Data]
         Aurora[(Amazon Aurora MySQL)]
      end

      CloudWatch[Amazon CloudWatch]
   end

   UserAccount --> ALB
   ALB --> Service
   Service --> Pod1
   Service --> Pod2
   Service --> Pod3
   Pod1 --> Aurora
   Pod2 --> Aurora
   Pod3 --> Aurora
   Pod1 -. Logs .-> CloudWatch
   Pod2 -. Logs .-> CloudWatch
   Pod3 -. Logs .-> CloudWatch
   APIServer -. Manage .-> NodeGroup
```

## 구성 요소 설명

1. **서버 실행: Amazon EKS**
   - Spring Boot 앱을 컨테이너로 여러 개 띄워서 운영합니다.
   - 트래픽이 늘어나면 실행 중인 앱 수를 늘려서 대응할 수 있습니다.
   - 여러 서버가 동시에 같은 사용자의 포인트를 건드릴 수 있기 때문에, 데이터 정합성은 DB 트랜잭션과 락으로 보장합니다.

2. **요청 분산: Application Load Balancer (ALB)**
   - 외부에서 들어오는 요청을 받아서 여러 앱 서버에 나눠줍니다.
   - HTTPS 처리도 여기서 담당합니다.

3. **내부 라우팅: Kubernetes Service**
   - 여러 앱 서버를 하나의 주소로 묶어서, 외부 요청이 자동으로 분산되도록 합니다.

4. **데이터베이스: Amazon Aurora MySQL**
   - 포인트 적립, 사용, 취소 등 모든 이력을 저장합니다.
   - 데이터 정합성이 중요한 시스템이라 모든 변경은 트랜잭션 단위로 처리합니다.
   - 장애가 발생해도 서비스가 유지되도록 이중화 구성을 합니다.

5. **모니터링: Amazon CloudWatch**
   - 앱 서버의 로그를 한 곳에 모아서 확인할 수 있습니다.
   - 서버 상태나 에러 발생 여부를 모니터링합니다.
